Expandable architecture notes

1. Product shape and guiding approach

The MVP deliberately has no public product UI: app/page.tsx returns a 404. The app is operated through email, database records, and developer scripts. This matches the core product idea: the owner can write an informal email request and get a draft invoice back for review.

The most important design choice is agent over rigid workflow. The input is free-form natural language, and future flows need revision/approval loops, so the app keeps business actions behind explicit tools while the LLM decides which tool to call next.

2. Inbound email security and idempotency

The inbound handler applies layered checks before it mutates state:

  • Webhook source: optional x-forwarded-for validation against Resend IPv4 and IPv6 ranges.
  • Webhook authenticity: Resend/Svix signature verification via RESEND_WEBHOOK_SECRET.
  • Sender authentication: DMARC must pass and header.from must match the sender domain.
  • Owner authorization: inbound from.email must match a row in users.
  • Rate limit: accepted owner emails are capped at 30 per hour.
  • Duplicate handling: an existing external_message_id makes Resend retries a no-op.
3. Durable context model

threads are conversation roots. messages are the append-only-ish history for user emails, assistant tool calls, and tool outputs. On every run, the agent reloads this DB history and rebuilds the LLM context from scratch.

User messages are converted to LLM messages that include both Subject: and Body:. Tool outputs are persisted as compact JSON, which makes the run replayable and debuggable without hidden process memory.

4. Agent loop and checkpointing

runAgentLoop is bounded by both step count and wall-clock seconds. Each unit of work is wrapped in runStep: load thread, load persona, compose prompt, call LLM, persist assistant, run tool, persist tool, mark status, or send reply.

The design convention from the docs is visible in code: do one awaitable thing, persist the result, then continue. That keeps the system ready for retries, job recovery, or eventual migration to a durable-execution platform.

load thread → render system prompt → LLM call
  → persist assistant message
  → execute requested tool(s)
  → persist tool result(s)
  → stop if terminal, otherwise loop
5. Tool set and invoice behavior

Read / write tools

  • search_client_db: owner-scoped fuzzy-ish lookup over company/contact/email/address/phone/notes.
  • manage_invoice: creates or updates draft invoices, calculates totals, writes line items, and assigns YYYY-XXXX invoice numbers.

Terminal tools

  • send_invoice_for_review: renders/reuses a PDF and emails the owner. Stops the run.
  • request_clarification: emails a focused follow-up question. Stops the run.

The terminal-tool convention is crucial: after an email is sent to the owner, the agent does not call the LLM again until the owner replies.

6. Data model at a glance

The schema is intentionally compact:

  • Identity: users and clients.
  • Conversations: threads and messages.
  • Work queue: jobs.
  • Invoice domain: invoices and line_items.
  • Runtime configuration: settings, including model name and system prompt.

Foreign keys are scoped to users where appropriate, clients cascade with users, line items cascade with invoices, and invoice numbers are unique per user.

7. Testing and operational status

The project has focused unit tests around the most important seams: inbound webhook behavior, thread resolution, Resend normalization, OpenAI Responses mapping, terminal agent behavior, and job status transitions.

  • Validated: production inbound email persistence, webhook signature rejection, DB-backed prompt, missing-client clarification email.
  • Still needs proof: full seeded-client happy path, reply/revision/approval flow, cron recovery, real production admin/auth flows.
  • Important deployment policy: Vercel Git auto-deploys are disabled; deploy manually with the Vercel CLI.
